# AliasVault on Podman with Quadlets

A complete guide for deploying [AliasVault](https://github.com/aliasvault/aliasvault) using **Podman Quadlets** (systemd-native container management) on RHEL-based systems (Rocky Linux, AlmaLinux, RHEL 9+).

This guide covers the multi-container setup behind an **existing reverse proxy** with your own TLS certificates — no built-in Let's Encrypt required.

> **Why Podman instead of Docker?**  
> Podman is daemonless, rootful or rootless, and integrates natively with systemd via Quadlets. On RHEL-based systems it's the default container runtime and a natural fit for production infrastructure.

## 1. Architecture Overview

AliasVault's official multi-container setup consists of 7 containers. In Docker Compose, these communicate over a shared Docker network using DNS-based service names (e.g. `postgres`, `api`). 

**With Podman Pods, networking works differently:** all containers in a Pod share the same network namespace — they all see each other on `localhost`. There is no built-in DNS resolution by container name.

The solution is to use `AddHost` entries in the Pod unit file to map service names to `127.0.0.1`. This way the application's hardcoded hostnames (like `Host=postgres` in connection strings) resolve correctly without modifying any container images.

```
                    ┌─────────────────────────────────────────────┐
  Internet          │  Podman Pod: aliasvault                     │
      │             │                                             │
      ▼             │  ┌──────────────┐    ┌──────────────────┐   │
  ┌────────┐        │  │reverse-proxy │───▶│   client :80     │   │
  │ nginx  │──443──▶│  │  :80/:443    │───▶│   api    :3001   │   │
  │ (host) │        │  └──────────────┘───▶│   admin  :3002   │   │
  └────────┘        │                      └──────────────────┘   │
      │             │  ┌──────────────┐    ┌──────────────────┐   │
      └──11443:443─▶│  │   postgres   │    │   smtp           │   │
                    │  │   :5432      │    │   task-runner    │   │
    SMTP :25/587───▶│  └──────────────┘    └──────────────────┘   │
                    └─────────────────────────────────────────────┘
```

| Container | Image | Purpose |
|-----------|-------|---------|
| `reverse-proxy` | `ghcr.io/aliasvault/reverse-proxy` | Internal nginx, terminates Pod-internal TLS |
| `api` | `ghcr.io/aliasvault/api` | REST API backend (.NET 10, port 3001) |
| `admin` | `ghcr.io/aliasvault/admin` | Admin panel (.NET 10, port 3002) |
| `client` | `ghcr.io/aliasvault/client` | Blazor WASM frontend (nginx) |
| `smtp` | `ghcr.io/aliasvault/smtp` | Inbound mail server for email aliases |
| `task-runner` | `ghcr.io/aliasvault/task-runner` | Background jobs |
| `postgres` | `docker.io/library/postgres:16-alpine` | PostgreSQL 16 database |

**External nginx** (running on the host) handles public TLS termination and proxies to the Pod on port `11443`.

## 2. Prerequisites

- RHEL 9 / Rocky Linux 9 / AlmaLinux 9 (or compatible)
- Podman ≥ 4.4 (for Quadlet support)
- systemd ≥ 252
- nginx (host-level, for TLS termination)
- Valid TLS certificate for your domain
- Ports 80, 443, 25, 587 available (or mapped to free ports)
- DNS control for your alias email domain(s)

```bash
# Verify Podman version supports Quadlets
podman --version   # should be >= 4.4

# Quadlet generator should exist
ls /usr/lib/systemd/system-generators/podman-system-generator
```

## 3. Directory Structure & Secrets

### 3.1 Create directories

```bash
mkdir -p /srv/aliasvault/{database,certificates/ssl,certificates/letsencrypt,logs,secrets}
```

```
/srv/aliasvault/
├── .env                        # Main configuration file
├── certificates/
│   ├── ssl/                    # Self-signed certs (auto-generated by reverse-proxy container)
│   └── letsencrypt/            # Unused if external nginx handles TLS
├── database/                   # PostgreSQL data files
├── logs/                       # Application logs from all containers
└── secrets/                    # Secret files (chmod 600)
    ├── jwt_key                 # JWT signing key (base64, 32 bytes)
    ├── postgres_password       # PostgreSQL password (base64, 32 bytes)
    ├── data_protection_cert_pass  # ASP.NET Data Protection key passphrase
    └── admin_password          # Admin password (plaintext, for reference)
```

### 3.2 Generate secrets

```bash
openssl rand -base64 32 > /srv/aliasvault/secrets/postgres_password
openssl rand -base64 32 > /srv/aliasvault/secrets/jwt_key
openssl rand -base64 32 > /srv/aliasvault/secrets/data_protection_cert_pass
openssl rand -base64 16 > /srv/aliasvault/secrets/admin_password
chmod 600 /srv/aliasvault/secrets/*
```

> **Why are secrets stored as files?**  
> AliasVault's API reads `jwt_key` and `postgres_password` **directly from the filesystem** at `/secrets/` — not from environment variables. The mount path `/secrets` is hardcoded in the application. Using `/run/secrets` (the Docker Swarm convention) will cause the API to start but fail on the first authenticated request with a `KeyNotFoundException`.

## 4. Environment Configuration

### 4.1 Create `.env`

```bash
PG_PASS=$(cat /srv/aliasvault/secrets/postgres_password)
JWT=$(cat /srv/aliasvault/secrets/jwt_key)
CERT_PASS=$(cat /srv/aliasvault/secrets/data_protection_cert_pass)

cat > /srv/aliasvault/.env << EOF
# General
HOSTNAME=vault.yourdomain.tld
HTTP_PORT=80
HTTPS_PORT=443
SMTP_PORT=25
SMTP_SUBMISSION_PORT=587
FORCE_HTTPS_REDIRECT=true
LETSENCRYPT_ENABLED=false
DEPLOYMENT_MODE=install

# Registration — set to false after creating your account(s)
PUBLIC_REGISTRATION_ENABLED=true

# Privacy
IP_LOGGING_ENABLED=false

# Email aliasing — comma-separated list of your alias domains
PRIVATE_EMAIL_DOMAINS=alias.yourdomain.tld

# SMTP TLS (enable once you have a cert for your alias domain)
SMTP_TLS_ENABLED=false

# Secrets (also read from /secrets/ files by the app)
POSTGRES_PASSWORD=${PG_PASS}
JWT_KEY=${JWT}
DATA_PROTECTION_CERT_PASS=${CERT_PASS}

# Database connection — IMPORTANT: use 127.0.0.1, not hostname 'postgres'
# The app builds the connection string from /secrets/postgres_password by default,
# but this env var takes precedence and correctly points to localhost (Pod networking).
ConnectionStrings__AliasServerDbContext=Host=127.0.0.1;Port=5432;Database=aliasvault;Username=aliasvault;Password=${PG_PASS}
EOF
```

### 4.2 Key configuration notes

| Variable | Value | Why |
|----------|-------|-----|
| `PUBLIC_REGISTRATION_ENABLED` | `true` initially, then `false` | Disable after creating your account(s) |
| `LETSENCRYPT_ENABLED` | `false` | External nginx handles TLS |
| `ConnectionStrings__AliasServerDbContext` | `Host=127.0.0.1` | Pod networking — all containers share localhost |
| `PRIVATE_EMAIL_DOMAINS` | your alias domain(s) | Enables custom email aliases |

## 5. Quadlet Unit Files

Quadlets are systemd unit files that Podman automatically translates into `podman run` commands. They live in `/etc/containers/systemd/`.

After creating or modifying any unit file, run:
```bash
systemctl daemon-reload
```

### 5.1 `aliasvault.pod`

> **Why `AddHost` entries belong in the Pod, not in individual containers:**  
> In a Podman Pod, all containers share one network namespace. `AddHost` entries written in a `.container` file are passed as `--add-host` to `podman run`, but when a container joins a pod, its network is the **pod's** network — so the entries have no effect. Placing them in the `.pod` file ensures they are set on the pod's infra container and are visible to all members.

```ini
[Unit]
Description=AliasVault Pod

[Pod]
PodName=aliasvault

# Map Docker Compose service names to localhost
# Required because all containers share the same network namespace in a Pod
# and there is no built-in DNS resolution by container name.
AddHost=client:127.0.0.1
AddHost=api:127.0.0.1
AddHost=admin:127.0.0.1
AddHost=postgres:127.0.0.1
AddHost=smtp:127.0.0.1

# Expose on non-standard ports to avoid conflicts with host services
# External nginx proxies 443 → 11443
PublishPort=11080:80
PublishPort=11443:443
PublishPort=25:25
PublishPort=587:587

[Install]
WantedBy=multi-user.target
```

### 5.2 `aliasvault-postgres.container`

> **Note on variable expansion in Quadlet files:**  
> Quadlet files do not support shell variable expansion. The `POSTGRES_PASSWORD` value must be written literally. Read it from the secrets file and write it directly — do not use `$()` substitution inside the unit file itself.
>
> **Security note:** writing `POSTGRES_PASSWORD` literally into the unit file is a deliberate trade-off for simplicity. See **Security Trade-offs → Postgres password in the container unit** for the `Secret=` alternative.

```ini
[Unit]
Description=AliasVault – Postgres
Requires=aliasvault-pod.service
After=aliasvault-pod.service

[Container]
Image=docker.io/library/postgres:16-alpine
ContainerName=aliasvault-postgres
Pod=aliasvault.pod
Environment=POSTGRES_DB=aliasvault
Environment=POSTGRES_USER=aliasvault
Environment=POSTGRES_PASSWORD=<contents of secrets/postgres_password>
Volume=/srv/aliasvault/database:/var/lib/postgresql/data:Z

[Service]
Restart=always
RestartSec=10
TimeoutStartSec=900

[Install]
WantedBy=multi-user.target
```

To write this file safely with the secret expanded:
```bash
printf '[Unit]\nDescription=AliasVault – Postgres\nRequires=aliasvault-pod.service\nAfter=aliasvault-pod.service\n\n[Container]\nImage=docker.io/library/postgres:16-alpine\nContainerName=aliasvault-postgres\nPod=aliasvault.pod\nEnvironment=POSTGRES_DB=aliasvault\nEnvironment=POSTGRES_USER=aliasvault\nEnvironment=POSTGRES_PASSWORD=%s\nVolume=/srv/aliasvault/database:/var/lib/postgresql/data:Z\n\n[Service]\nRestart=always\nRestartSec=10\nTimeoutStartSec=900\n\n[Install]\nWantedBy=multi-user.target\n' \
  "$(cat /srv/aliasvault/secrets/postgres_password)" \
  > /etc/containers/systemd/aliasvault-postgres.container
```

### 5.8 `aliasvault-reverse-proxy.container`

> **Why no `AddHost` here?**  
> The `AddHost` entries are in the Pod unit file (see section 5.1). Adding them here as well would be redundant and in a Pod context, container-level `--add-host` flags are ignored anyway since the container uses the pod's network namespace.

```ini
[Unit]
Description=AliasVault – Reverse Proxy
Requires=aliasvault-pod.service aliasvault-api.service aliasvault-client.service aliasvault-admin.service
After=aliasvault-pod.service aliasvault-api.service aliasvault-client.service aliasvault-admin.service

[Container]
Image=ghcr.io/aliasvault/reverse-proxy:latest
ContainerName=aliasvault-reverse-proxy
Pod=aliasvault.pod
# rw required — the container generates a self-signed cert on first start
Volume=/srv/aliasvault/certificates/ssl:/etc/nginx/ssl:rw,Z
Volume=/srv/aliasvault/certificates/letsencrypt:/etc/letsencrypt:ro,Z
Volume=/srv/aliasvault/logs:/logs:Z
EnvironmentFile=/srv/aliasvault/.env

[Service]
Restart=always
RestartSec=10
TimeoutStartSec=900

[Install]
WantedBy=multi-user.target
```

## 6. External Reverse Proxy (nginx)

The host nginx acts as the TLS frontend. It proxies HTTPS traffic to the Pod's internal reverse proxy on port `11443`. The Pod's internal TLS uses a self-signed certificate, so `proxy_ssl_verify off` is required.

`/etc/nginx/conf.d/aliasvault.conf`:

```nginx
upstream aliasvault {
  server 127.0.0.1:11443;
}

map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      "";
}

server {
    listen 80;
    listen [::]:80;
    server_name vault.yourdomain.tld;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    listen [::]:443 ssl;
    server_name vault.yourdomain.tld;

    ssl_certificate     /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;

    client_max_body_size 525M;

    location / {
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
        proxy_ssl_verify off;
        proxy_pass https://aliasvault;
    }
}
```

## 7. DNS & SMTP / Email Aliasing

AliasVault can receive email for custom domains and make them available as aliases. Each alias domain needs its own MX record pointing to your AliasVault server.

### 7.1 DNS records

For each alias domain (e.g. `alias.yourdomain.tld`):

```dns
; A record for the AliasVault host (must be A, not CNAME — MX cannot point to CNAME)
vault.yourdomain.tld.   IN A    <your-server-ip>

; Alias subdomain
alias.yourdomain.tld.   IN A    <your-server-ip>
alias.yourdomain.tld.   IN MX 10 vault.yourdomain.tld.
alias.yourdomain.tld.   IN TXT  "v=spf1 a:vault.yourdomain.tld ~all"
```

> **MX records must not point to a CNAME.** This is an RFC violation (RFC 2181 §10.3). Always use an A record as the MX target.

### 7.2 Multiple alias domains

Add all domains comma-separated in `.env`:

```bash
PRIVATE_EMAIL_DOMAINS=alias.domain1.tld,alias.domain2.tld
```

### 7.3 Firewall

Ensure the following ports are open in your firewall (firewalld, nftables, iptables, or cloud security group — depending on your environment):

| Port | Protocol | Purpose |
|------|----------|---------|
| 80 | TCP | HTTP (redirect to HTTPS) |
| 443 | TCP | HTTPS |
| 25 | TCP | SMTP (inbound mail for aliases) |
| 587 | TCP | SMTP submission |

## 7.4 SELinux (RHEL-based systems)

On SELinux-enforcing systems (RHEL, Rocky Linux, AlmaLinux), the host nginx process is confined and cannot by default connect to upstream ports on localhost. Without the following boolean, nginx will silently fail to proxy to the Pod with a `(13: Permission denied)` error in `/var/log/nginx/error.log`:

```bash
setsebool -P httpd_can_network_connect 1
```

The `:Z` volume mount flags in the container units already handle correct SELinux labeling for bind mounts — no additional `chcon` or policy changes are needed.

## 8. Starting the Stack

### Pull images first

On a fresh system, pull all images manually before starting the stack. This gives you direct feedback on download progress and avoids systemd timeouts during the first start:

```bash
podman pull docker.io/library/postgres:16-alpine
podman pull ghcr.io/aliasvault/api:latest
podman pull ghcr.io/aliasvault/admin:latest
podman pull ghcr.io/aliasvault/client:latest
podman pull ghcr.io/aliasvault/smtp:latest
podman pull ghcr.io/aliasvault/task-runner:latest
podman pull ghcr.io/aliasvault/reverse-proxy:latest
```

### Start the stack

```bash
systemctl daemon-reload

# Start in order — postgres needs time to initialize before the app containers
systemctl start aliasvault-pod.service

# Verify
systemctl status aliasvault-*.service --no-pager | grep -E '\.service|Active:'
```

### Enable on boot (Quadlet)

With Quadlets, you typically **do not** run `systemctl enable` on the generated `aliasvault-*.service` units.

Autostart is controlled via the **`[Install]`** section inside the Quadlet files (which this guide includes).

For servers, `multi-user.target` is the appropriate target:

```ini
[Install]
WantedBy=multi-user.target
```

After creating/updating Quadlets:

```bash
systemctl daemon-reload
systemctl start aliasvault-pod.service
```

If you try to systemctl enable the generated units anyway, systemd will usually complain about generated/transient units not being enable-able — that’s expected.


### First login

1. Navigate to `https://vault.yourdomain.tld`
2. Register your account (registration is enabled in `.env`)
3. After registration, disable public registration:

```bash
sed -i 's/^PUBLIC_REGISTRATION_ENABLED=true/PUBLIC_REGISTRATION_ENABLED=false/' /srv/aliasvault/.env
systemctl restart aliasvault-api.service
```

### Creating the initial admin account

AliasVault's admin panel requires an admin user in the `AdminUsers` table. In a Podman setup, the `installcli` container that Docker Compose normally uses for this does not work reliably. The workaround is to generate an ASP.NET Identity v3 compatible PBKDF2-SHA256 password hash in Python and insert the user directly via `psql`.

> **Why this specific hash format?**  
> ASP.NET Identity uses a binary format: a header byte (`0x01`), algorithm ID, iteration count, and salt length — followed by the salt and derived key, all base64-encoded. A plain bcrypt or SHA256 hash will be silently rejected at login.

```bash
PASS=$(cat /srv/aliasvault/secrets/admin_password)

HASH=$(python3 -c "
import hashlib, os, base64, struct
password = '$PASS'
salt = os.urandom(16)
iterations = 100000
dk = hashlib.pbkdf2_hmac('sha256', password.encode('utf-8'), salt, iterations, dklen=32)
header = struct.pack('>BIII', 0x01, 0x01, iterations, 16)
hash_bytes = header + salt + dk
print(base64.b64encode(hash_bytes).decode())
")

echo "Hash generated: ${HASH:0:20}..."

podman exec -it aliasvault-postgres psql -U aliasvault -d aliasvault -c \
  "INSERT INTO \"AdminUsers\" (
     \"Id\", \"UserName\", \"NormalizedUserName\",
     \"Email\", \"NormalizedEmail\", \"EmailConfirmed\",
     \"PasswordHash\", \"SecurityStamp\", \"ConcurrencyStamp\",
     \"PhoneNumberConfirmed\", \"TwoFactorEnabled\", \"LockoutEnabled\", \"AccessFailedCount\"
   ) VALUES (
     gen_random_uuid(), 'admin', 'ADMIN',
     'admin@localhost', 'ADMIN@LOCALHOST', true,
     '$HASH', gen_random_uuid()::text, gen_random_uuid()::text,
     false, false, false, 0
   );"
```

The admin panel is then accessible at `https://vault.yourdomain.tld/admin`.

## 10. Known Pitfalls

These are the non-obvious issues you will likely hit when adapting the Docker Compose setup to Podman Pods.

| Issue | Symptom | Fix |
|-------|---------|-----|
| Wrong secrets mount path | API starts but crashes with `KeyNotFoundException: JWT key file not found at /secrets/jwt_key` on first request | Mount must be `:/secrets`, not `:/run/secrets` |
| `AddHost` in container unit | DNS names like `postgres` don't resolve inside the Pod | Move all `AddHost` entries to the `.pod` unit file |
| MX points to CNAME | Mail delivery fails / MX lookup warnings | MX target must be an A record |
| Variable expansion in Quadlet | `POSTGRES_PASSWORD=$(cat ...)` written literally, not expanded | Use `printf` or editor to write the literal value |
| Postgres password in unit file | Password exists in plaintext in `/etc/containers/systemd/aliasvault-postgres.container` | This is a deliberate trade-off for simplicity. See **Security Trade-offs → Postgres password in the container unit** for a `Secret=` alternative |
| `ssl` volume must be `rw` | Reverse proxy container fails to start | It generates a self-signed cert on first start — needs write access |
| First boot / image pull timeout | systemd kills containers after 90s before images finish pulling | `TimeoutStartSec=900` in `[Service]` (already set in this guide's units) |

## 11. Optional: External PostgreSQL

If you want to use an existing PostgreSQL instance instead of the bundled container:

1. Remove `aliasvault-postgres.container` and its systemd dependencies
2. Update `.env`:

```bash
ConnectionStrings__AliasServerDbContext=Host=<your-pg-host>;Port=5432;Database=aliasvault;Username=aliasvault;Password=<password>
```

3. Update the Pod's `AddHost` to point `postgres` at your external host:

```ini
AddHost=postgres:<your-pg-host-ip>
```

Or simply use the IP directly in the connection string and remove the `postgres` `AddHost` entry entirely.

Make sure the database `aliasvault` exists and the user has full privileges before starting the API container — it runs migrations on startup.

## Security Trade-offs

### Postgres password in the container unit

This guide stores the PostgreSQL password as a literal value in `aliasvault-postgres.container` (see section 5.2). Podman Quadlets do support a native `Secret=` directive that avoids this, but it comes with trade-offs:

**The case for `Secret=`:** The password does not appear in a root-readable config file. It is managed via `podman secret create` and stored in Podman's secret store.

**Why this guide uses a literal value instead:**
- The postgres container password already exists on disk as `/srv/aliasvault/secrets/postgres_password` (chmod 600, root-only) — the security boundary is the same filesystem permission either way.
- Podman Secrets add a second secret management system alongside the `/secrets/` file mounts the application already requires. For a single-admin self-hosted setup, the added complexity outweighs the benefit.
- The guide is intentionally explicit and debuggable. A `Secret=` setup that fails silently is harder to troubleshoot for someone new to Quadlets.

If you prefer `podman secret` for the Postgres password:

```bash
podman secret create aliasvault-postgres-password /srv/aliasvault/secrets/postgres_password
```

Then replace the `Environment=POSTGRES_PASSWORD=...` line in `aliasvault-postgres.container` with:

```ini
Secret=aliasvault-postgres-password,type=env,target=POSTGRES_PASSWORD
```

## Skeleton Files

The `skeleton/` directory in this repository contains ready-to-use versions of all configuration files referenced in this guide:

```
skeleton/
├── srv/aliasvault/
│   ├── .env.example          # All variables with placeholders — copy to .env and fill in
│   └── secrets/.gitkeep      # Placeholder — never commit actual secret files
└── etc/containers/systemd/
    ├── aliasvault.pod
    ├── aliasvault-postgres.container
    ├── aliasvault-api.container
    ├── aliasvault-admin.container
    ├── aliasvault-client.container
    ├── aliasvault-smtp.container
    ├── aliasvault-task-runner.container
    └── aliasvault-reverse-proxy.container
```

Copy the Quadlet files to `/etc/containers/systemd/` and `.env.example` to `/srv/aliasvault/.env`, then follow the guide.

> **Before starting the stack:** Remove the `.gitkeep` placeholder from the database directory — PostgreSQL will refuse to initialize if the data directory contains any files, including hidden ones:
> ```bash
> rm /srv/aliasvault/database/.gitkeep
> ```
